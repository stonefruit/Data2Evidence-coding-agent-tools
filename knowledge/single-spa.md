# single-spa Portal Integration

## Read When

- A task mentions single-spa, micro-frontends, import maps, SystemJS, `import-map-overrides`, or plugin apps.
- Work changes portal plugin loading, researcher/system-admin/ETL routes, or `type: "app"` plugin configuration.
- A single-spa child app fails to mount, mounts in the wrong container, receives stale props, or ignores a local override URL.

## Summary

D2E uses single-spa to load selected UI plugins as independently built micro-frontends inside the portal. The portal remains the shell: it reads plugin metadata, registers `type: "app"` plugins with single-spa, loads lifecycle modules through SystemJS, passes portal context as custom props, and renders the DOM containers child apps mount into.

## Facts

- The current portal registry lives in `repos/Data2Evidence/plugins/ui/apps/portal/src/singleSpa/`. `singleSpaRegistry.ts` wraps `registerApplication`, `start`, `unregisterApplication`, module caching, custom prop updates, and `window.System.import`.
- Plugin metadata distinguishes single-spa apps with `type: "app"`. Researcher, System Admin, and ETL containers split plugin lists into single-spa apps and legacy plugins, then render `SingleSpaAppContainer` for app plugins.
- Researcher apps are registered by `ResearcherStudyPluginRenderer.tsx`; System Admin and ETL apps use `SystemAdminSingleSpaPluginRenderer.tsx`. Both generate stable app ids from `pluginPath`, compute a base path from `env.PUBLIC_URL`, pass auth/user/study/system context as `customProps`, call `startSingleSpa({ urlRerouteOnly: true })`, update props through `updateCustomProps`, and unregister on unmount.
- Route activation is handled by `createActivityFunction(basePath, autoMount)`. Without `autoMount`, an app is active when `location.pathname` exactly matches the base path or starts with `basePath + "/"`; with `autoMount`, the app is always active while registered.
- The portal container id format is `single-spa-application:<appId>`, generated by `generateContainerId`. React child apps should use `single-spa-react` and a `domElementGetter` that mounts into the `containerId` prop.
- The portal pre-renders containers for single-spa apps in Researcher, System Admin, and ETL and toggles visibility by route. Loading state comes from single-spa status checks in `SingleSpaAppContainer`, not from React Router routes alone.
- Child apps such as Concept Sets and Wizards export `bootstrap`, `mount`, and `unmount` from `src/lifecycles.tsx`, and their Vite configs build SystemJS library output named `lifecycles.js`.
- `import-map-overrides` is loaded by the portal and exposed through the hidden UI in `public/index.html`, shown when `localStorage.devtools` is set. Because this portal uses SystemJS compatibility behavior rather than native browser import maps, override resolution is manually checked in `singleSpa/overrideUtils.ts` and the plugin loader.
- `vue-mri-ui-lib` also contains a single-spa registry in `src/utils/AppRegistry.ts` and container rendering in `RootLayout.vue`. Treat it as a separate shell-style integration when working in the MRI UI library rather than assuming the portal registry owns those apps.

## Recheck When

- The portal upgrades SystemJS, changes import-map handling, or removes the manual `import-map-overrides` bridge.
- Plugin metadata changes fields such as `pluginPath`, `type`, `route`, or `autoMount`.
- single-spa registration, generated app ids, container id format, or custom prop payloads change.
- A child app changes its build format, lifecycle export shape, or `domElementGetter` behavior.
